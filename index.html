디자인 대상의 전환,
앱에서 에이전트의 시대로

사용자가 기능을 찾아다니는 시대에서 에이전트가 목적을 대신 수행하는 시대로

Shift 01
UI to Intent
메뉴 중심에서 의도 중심으로
App · 앱
사용자가 메뉴를 탐색하고 기능을 하나씩
실행하는 도구함 (유저가 길을 알아야 함)
Agent · 에이전트
사용자의 추상적인 의도를 이해하고
최적의 경로를 스스로 생성 (에이전트가 길을 찾음)
Insight How to use의 정교함보다 What to do의 정확성이 핵심
Shift 02
Feature to Goal
기능의 나열에서 과업의 완결로
App · 앱
'송금하기', '장바구니 담기' 같은 파편화된 기능을
제공. 최종 결과는 유저의 노동으로 완성
Agent · 에이전트
'이번 주말 여행 준비해 줘' 하나의 목표를 이해,
여러 기능을 엮어 과업을 완결하는 존재
Insight 성공지표가 '체류시간(Retention)'에서 '과업 성공률(Completion Rate)'로 이동
Shift 03
Passive to Proactive
수동적 반응에서 선제적 대응으로
App · 앱
사용자가 아이콘을 눌러야만 움직이는
명령에 의존하는 수동적 존재
Agent · 에이전트
상황과 맥락을 모니터링하다가 필요한 순간에
먼저 말을 걸거나 액션을 제안
Insight 인터랙션 트리거가 유저의 손가락이 아닌 에이전트의 상황 판단

에이전트 존재의 설계도

에이전트라는 봇에 스킬과 메모리를 갈아끼우면, 특정 가치를 창출하는 한 명의 에이전트가 된다.

01
Head
두뇌 LLM
사용자 쿼리를 이해하고 "어떤 사이트에서 무엇을 클릭해야겠다"를 추론한다.
02
Antenna
감지 Perception
가격 변동, 재고 알림, 사용자 메시지 — 외부 신호를 끊임없이 감지한다.
03
Body
액추에이터 Actuator
뇌의 명령을 받아 실제 브라우저를 열고 로그인하고 클릭하고 결제한다.
04
Wrench
스킬 Skill
"최저가를 찾을 때 배송비까지 포함하라" — 판단 로직을 도구로 갈아끼운다.
05
Disc
메모리 Memory
신용카드 정보, 배송지 주소, 선호 브랜드 — 행동에 필요한 맥락을 보유한다.

메모리 아키텍처: 에이전트의 기억법

에이전트는 어떻게 기억하고, 무엇을 기억하는가

A. 메모리 타임라인 FLOW
B. 장기기억의 3가지 타입 TYPES
01
SESSION
현재 세션 — 단기기억
사용자와 주고받는 대화·상태가 컨텍스트 윈도우에 적재된다. 빠르지만 세션 종료 시 휘발.
In-context · Working Memory
02
EXTERNAL STORE
외부 저장 — 장기기억
중요 정보는 벡터·지식 그래프·룰 스토어에 영속화된다. 세션을 넘어 누적·재사용 가능한 기억의 풀.
Vector DB · Knowledge Base
03
RETRIEVAL
검색 — RAG 주입
새 입력이 들어오면 관련 기억을 검색해 컨텍스트에 주입한다. 응답은 항상 현재 + 누적 기억의 합산.
RAG · Context Injection
↻ 매 턴마다 세션 → 저장 → 검색 사이클이 반복되어, 에이전트는 시간이 갈수록 사용자에 대해 더 잘 알게 된다.
EPISODIC MEMORY
경험 기억
과거 상호작용의 기록
WHAT IT STORES
언제·무엇을·어떻게 했는지 — 시간 순으로 누적되는 사용자별 행동 이력.
EXAMPLE
지난번에도 강남으로 배송 받았다
SEMANTIC MEMORY
사실 기억
사용자 정보 · 선호 · 도메인 지식
WHAT IT STORES
변하지 않는 사실과 선호. 사용자의 프로필, 취향, 도메인 상수 같은 영속 지식.
EXAMPLE
이 사용자는 나이키를 선호한다
PROCEDURAL MEMORY
절차 기억
반복 워크플로우 · 판단 로직
WHAT IT STORES
"이런 상황에서는 이렇게 처리한다"는 룰과 휴리스틱 — 행동 패턴 자체.
EXAMPLE
최저가 탐색 시 배송비 포함 계산
왼쪽: 시간축으로 본 메모리 흐름 — 세션의 단기기억은 외부 저장소로 영속화되고, 검색 단계에서 RAG로 다시 주입된다. | 오른쪽: 외부 저장소에 쌓이는 장기기억은 성격에 따라 3가지 타입으로 분류된다.
2.5

OpenClaw 와 Agentic AI Framework

에이전트를 움직이게 하는 두 가지 방식의 프레임워크

프레임워크는 LLM의 추론을 실제 환경에서 작동시키는 실행 레이어다. 작동 환경이 있어야 LLM은 비로소 에이전트가 된다.

Shared Principles
01
LLM은 두뇌, 프레임워크는 신체와 환경.
02
메모리·스킬·액추에이터가 조합되어야 에이전트가 완성된다.
03

유동적 인터페이스 Generative UI
에이전트가 답변의 성격과 의도에 따라 UI를 스스로 결정하고 생성하는 인터페이스 패러다임
기존 앱 UX
미리 설계된 화면 레이아웃과 확정된 에셋 라이브러리 안에서만 작동. 사용자가 할 수 있는 행동과 볼 수 있는 정보가 설계 시점에 모두 확정됨. 설계되지 않은 상황은 존재하지 않음.
에이전트 UX
답변의 성격·의도·데이터 구조에 따라 그때그때 가장 적합한 UI를 에이전트가 판단하고 생성. 사전에 모든 상황을 설계할 수 없기 때문에 UI 자체가 동적으로 만들어져야 함.
방식
설명
적합한 상황
Rule-based Generation
상황별 최적 UI 룰이 사전 정의되어 있고 에이전트가 룰을 참조해 UI 결정
반복적·예측 가능한 답변 유형
Library-based Generation
UI 컴포넌트 라이브러리에서 에이전트가 적합한 컴포넌트를 조합해 생성
구조화된 데이터, 비교·선택이 필요한 상황
From Scratch Generation
기존 룰이나 라이브러리로 커버되지 않는 예외적 상황에서 에이전트가 UI를 직접 설계
경향을 벗어난 복합적·특수한 답변
DESIGN.md — 에이전트를 위한 디자인 시스템
↑ Rule-based Generation의 실제 구현체
기존 방식
에이전트에게 매번 브랜드 컬러·폰트·컴포넌트를 설명해야 함. 툴을 바꿀 때마다, 새 대화를 시작할 때마다 처음부터 다시.
DESIGN.md 방식
파일 하나로 끝. YAML 토큰(정확한 값) + 마크다운 텍스트(왜 이 값인지 이유)를 함께 담아 에이전트가 자동으로 읽고 적용.

Google Labs가 2026년 4월 오픈소스로 공개. Claude Code, Cursor, Copilot 모두 지원.
