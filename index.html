1. UI to Intent
메뉴 중심에서 의도 중심으로
사용자가 메뉴를 탐색하고 기능을 하나씩 실행하는 도구함 (유저가 길을 알아야 함)
사용자의 추상적인 의도를 이해하고 최적의 경로를 스스로 생성 (에이전트가 길을 찾음)
How to use의 정교함보다 What to do의 정확성이 핵심
2. Feature to Goal
기능의 나열에서 과업의 완결로
'송금하기', '장바구니 담기' 같은 파편화된 기능 제공. 최종 결과는 유저의 노동으로 완성
'이번 주말 여행 준비해 줘' 하나의 목표를 이해, 여러 기능을 엮어 과업을 완결
성공지표가 '체류시간(Retention)'에서 '과업 성공률(Completion Rate)'로 이동
3. Passive to Proactive
수동적 반응에서 선제적 대응으로
사용자가 아이콘을 눌러야만 움직이는 수동적 존재
상황과 맥락을 모니터링하다가 필요할 때 먼저 말을 걸거나 액션
인터랙션 트리거가 유저의 손가락이 아닌 에이전트의 상황 판단

1. 리서치
Synthesis over Collection
유저 인터뷰, 데스크 리서치로 파편화된 정보를 직접 수집하는 데 많은 시간 할애
에이전트가 방대한 데이터를 실시간 요약·대조. 디자이너는 새로운 인사이트를 '합성'
정보 탐색의 물리적 시간이 줄어들고 질문의 질이 리서치의 성패를 결정
2. 아이데이션
Co-creation over Ideation
화이트보드 앞에서 포스트잇을 붙이며 스스로 아이디어를 짜냄
코딩/이미지/텍스트 에이전트와 대화하며 수많은 시안을 초단위로 검토
무에서 유를 만드는 고통 대신, 제안받은 가능성 사이에서 최적의 논리를 선택하고 정교화
3. 프로토타이핑
Implementation over Description
실제처럼 보이는 가짜 화면을 그려 개발자에게 설명 (Draw to show)
코딩 에이전트와 함께 실제로 돌아가는 지능의 로직을 즉시 코드로 구현 (Build to test)
시각적 묘사에 머물던 프로토타이핑이 작동하는 로직 검증으로 격상. 그림이 아닌 '작동하는 실체'로 소통
4. 검증
Evaluation over Observation
사용자가 버튼을 잘 누르는지 뒤에서 지켜보는 관찰자
수만 개의 가상 시나리오를 통과시켜 답변 품질을 점수화, 로직을 수정하는 시스템 평가자
한두 명의 유저 반응을 보는 것을 넘어 시뮬레이션을 통해 테스트하고 튜닝

에이전트 프레임워크란 무엇인가
프레임워크는 LLM의 추론을 실제 환경에서 작동시키는 실행 레이어다.

LLM 단독으로는 단일 응답을 넘어선 과업을 수행할 수 없다 — 작동 환경이 있어야 비로소 에이전트가 된다.
두 프레임워크가 공유하는 설계 원칙
  • LLM은 두뇌, 프레임워크는 신체와 환경
  • 메모리·스킬·액추에이터가 조합되어야 에이전트가 완성됨
  • 오케스트레이션 레이어가 추론·도구 호출·메모리 접근을 하나의 작업 흐름으로 엮는다

기존 앱 UX
미리 설계된 화면 레이아웃과 확정된 에셋 라이브러리 안에서만 작동. 사용자가 할 수 있는 행동과 볼 수 있는 정보가 설계 시점에 모두 확정됨. 설계되지 않은 상황은 존재하지 않음.
에이전트 UX
답변의 성격·의도·데이터 구조에 따라 그때그때 가장 적합한 UI를 에이전트가 판단하고 생성. 사전에 모든 상황을 설계할 수 없기 때문에 UI 자체가 동적으로 만들어져야 함.

Before
프로토타이핑의 한계가 Figma였음. 실제로 작동하는 걸 만들려면 개발자에게 요청해야 했고, 그 사이에 아이디어가 죽거나 왜곡됨. 코드는 내 영역이 아니었음.
