# Part 4. 실무 프로세스: 일하는 방식의 변화와 산출물
## Chap 4-C. 협업 프로토콜: 개발자와 디자이너 사이의 달라진 소통 방식

**챕터 간지 핵심 메시지:** "개발자-에이전트-디자이너, 함께 일하기"

## 4.7 정적 가이드에서 실행 가능한 로직으로

**도입 카피:** "Implementation over Description – 묘사를 넘어 구현으로 소통하기"

**페이지 구성:** 3단 구성 (핵심 변화 + 소통 방식 + 산출물)

**핵심 변화**
Figma 픽셀·정적 화면을 넘겨주던 방식 → 디자인 단계에서 에이전트와 함께 검증한 로직을 넘겨주는 방식

**소통 방식**
"이 화면은 이렇게 보여야 해요" 대신 "에이전트와 함께 이 로직으로 PoC를 돌려봤을 때 유저 반응이 가장 좋았어요. 이 코드 스니펫을 참고해 주세요"

**산출물**
디자인 가이드라인 + 에이전트 협업으로 도출된 로직 명세 및 참조 코드

---

## 4.8 시각적 검수에서 시스템적 검증으로

**도입 카피:** "Evaluation over Observation – 관찰을 넘어 평가로"

**페이지 구성:** 3단 구성 (핵심 변화 + 소통 방식 + 산출물)

**핵심 변화**
개발 후 눈으로 디자인 정합성을 보던 관찰 중심 QA → 에이전트를 활용한 자동화된 시뮬레이션 기반 검증

**소통 방식**
디자이너·개발자가 평가 프레임워크를 공유. 에이전트가 수만 번의 시나리오를 돌려 발견한 엣지 케이스를 함께 공유하며 시각적 완성도뿐 아니라 성능·신뢰도를 함께 튜닝

**산출물**
디자인 QA 결과서 + 에이전트 시뮬레이션 기반 품질 성적표 및 예외 케이스 리포트

---

## 4.9 레이아웃 전달에서 구조적 페어링으로

**도입 카피:** "Co-creation over Handoff – 이관을 넘어 공동 설계로"

**페이지 구성:** 3단 구성 (핵심 변화 + 소통 방식 + 산출물)

**핵심 변화**
디자인 완료 후 개발팀에 일방적으로 이관하던 방식 → 에이전트를 매개로 기술적 구조를 함께 설계하는 파트너십

**소통 방식**
디자이너가 에이전트와 PoC를 만드는 과정에서 얻은 기술적 이해를 바탕으로 API 효율성·데이터 구조에 대해 개발자와 대등하게 토론. 예: "이 데이터를 불러올 때 에이전트가 이런 병목을 겪던데, API 구조를 이렇게 가져가면 어떨까요?"

**산출물**
최종 디자인 에셋 + 디자이너-개발자 공동 설계 API/데이터 스키마 규격서
