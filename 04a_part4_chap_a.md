## 4.2 하네스 엔지니어링

**도입 카피:** "Human-in-the-loop 게이트 조건과 행동 가이드레일 정의"

**페이지 레이아웃:** 4.1과 동일한 캡처 슬라이드 형식. Z패턴 비대칭 레이아웃 — 좌측은 콘텐츠 위 / 콜아웃 아래, 우측은 콜아웃 위 / 콘텐츠 아래로 회전 흐름.

**슬라이드 상단 좌측 외곽 (메타 주석, 한 줄):**
'"사용자가 직접 고를 수 있게 해줘", "출처 없는 말은 하지 마" 같은 단편적인 요구사항도 Claude가 HITL 게이트와 행동 원칙으로 변환해서 시스템 프롬프트에 반영.'

**캡처 슬라이드 헤더:**
- 소제목 (작은 그레이): System Prompt
- 슬라이드 제목: Trend Research 에이전트 만들기

### 좌측 컬럼 (콘텐츠 상단 정렬, HITL 게이트)

각 콘텐츠 위에 메타 태그 "(시스템 프롬프트 중략)" — 작은 회색 이탤릭, 제목 바로 위에 붙여서 (4.1로부터의 연속성 표시)

**Step 2. 토픽 요약 제시 — Human 선택 게이트** (이 라벨이 곧 좌측 섹션 제목)

선정된 3개 토픽을 각각 3-4문장으로 요약해 사용자에게 제시.
요약에는 토픽의 배경, 왜 지금 화제인지, UX적 의미가 한눈에 보여야 함.

```
이번 주 UX/디자인 분야 화제 토픽 3개를 정리했습니다.
리서치를 진행할 토픽 1개를 선택해 주세요.

① {토픽 제목}
{배경 및 왜 지금 화제인지 2문장}
{UX적 의미 1-2문장}
주요 출처: {출처명}

② ③ 동일 구조

①②③ 중 하나를 선택해 주시면 해당 토픽으로
심층 인사이트 리포트를 작성합니다.
```

사용자가 번호나 토픽명으로 선택 → Step 3으로 진행
"다시 찾아줘" 또는 마음에 드는 것이 없을 경우 → Step 1을 다른 쿼리로 재실행

### 우측 컬럼 (콘텐츠 하단 정렬, 공통 행동 원칙)

위에 메타 태그 "(시스템 프롬프트 중략)" + 컬럼 소제목 "공통 행동 원칙" + 4개 항목

- **근거 없는 주장 금지:** 모든 인사이트에는 출처(아티클 URL, 플랫폼, 날짜) 반드시 포함. "~인 것 같습니다" 같은 추측성 표현 사용 금지
- **도메인 특수성 우선:** 일반적인 UX 트렌드보다 해당 도메인에 특화된 인사이트 우선. 예) 헬스케어라면 의료 규제, 고령자 접근성, EMR 연동 맥락 반드시 고려
- **Human 게이트 준수:** 분석 완료 후 바로 다음 모듈로 넘기지 않음. 반드시 사용자에게 결과 제시 후 승인 또는 보정 요청. 사용자가 "승인" 또는 "다음으로" → next_module_ready:true 상태로 JSON 확정
- **언어:** 사용자가 한국어로 대화하면 한국어로 응답. JSON 산출물 내 서술 필드도 한국어로 작성

### 오버레이 + 표시선 + 말풍선 (총 2개, 번호 배지)

**1번** (좌하단 빈 영역, 슬라이드 안쪽): "HITL 게이트 조건 정의해줘"
- HITL 오버레이 하단 변 중앙에서 1번 말풍선 좌상단으로 사선 표시선

**2번** (슬라이드 우측 외곽선 걸침, 상단): "행동 가이드레일 정의해줘"
- 행동 원칙 오버레이 상단 중앙에서 2번 말풍선 좌하단으로 사선 표시선

좌측 HITL 오버레이와 우측 행동 원칙 오버레이는 슬라이드 중간 영역에서 세로로 일부 중첩되어 다이내믹한 리듬 형성.

## 4.4 디자인시스템의 진화

**도입 카피:** "컴포넌트 라이브러리에서 에이전트가 소비하는 규칙 시스템으로"

**페이지 구성:** 상단 2블록 대비 + 중단 테이블 + 하단 핵심 메시지

### 상단: 기존 디자인 시스템 vs 에이전트 시대 디자인 시스템 (2블록 대비)

**기존 디자인 시스템**
디자이너가 미리 만들어둔 컴포넌트·색상·타이포그래피 라이브러리. 사람이 골라서 쓰는 정적 에셋 모음. 잘 만들어도 에이전트는 읽지 못함.

**에이전트 시대 디자인 시스템**
에이전트가 UI를 생성할 때 참조하는 규칙·판단 기준·컴포넌트 풀. 사람이 아닌 에이전트가 소비하는 시스템. 컴포넌트뿐 아니라 "어떤 상황에 어떤 UI를 써야 하는가"의 판단 로직까지 포함.

### 중단: 디자인 시스템에 새롭게 추가되는 레이어 (테이블)

*1열과 2열은 대등한 관계 (제목/상세 위계 아님). 좌우 셀 모두 동일한 폰트 사이즈·굵기로 표현.

| 기존 레이어 | 추가되는 레이어 |
|------------|---------------|
| 컴포넌트 라이브러리 | UI 생성 룰 (어떤 상황에 어떤 컴포넌트를 쓸지 판단 기준) |
| 컬러·타이포·그리드 토큰 | 에이전트용 컨텍스트 토큰 (상황·의도·데이터 타입별 매핑) |
| 사용 가이드라인 | 예외 케이스 정의 (룰 밖의 상황에서 From Scratch 생성 기준) |
| Figma 파일 | 에이전트가 읽을 수 있는 구조화된 스펙 (JSON, SKILL.md 등) |

### 하단: 디자이너 역할의 전환
화면을 그리는 것이 아닌 UI가 생성되는 기준과 문법을 설계. 디자인 시스템이 사람을 위한 가이드북에서 에이전트를 위한 판단 엔진으로 진화.
