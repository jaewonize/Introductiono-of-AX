## 4.5 디자인 자동화 루틴

**도입 카피:** "반복하는 일은 에이전트에게, 판단하는 일은 디자이너에게"

**페이지 구성:** 상단 1블록 + 중단 테이블 + 하단 원칙

### 상단: 자동화 가능한 디자인 업무의 속성 (1블록)
반복적이고, 규칙이 있고, 대량 처리가 필요한 일이 자동화 적합. 디자이너는 기준을 세우고 에이전트는 그 기준대로 실행.

### 중단: 디자인 프로세스별 자동화 루틴 (테이블)

| 프로세스 | 자동화 가능한 업무 | 에이전트에게 시키는 방식 |
|---------|-----------------|----------------------|
| **리서치** | 경쟁사 UI 수집, 트렌드 키워드 모니터링, 레퍼런스 정리 | "이 카테고리 앱 10개 UI 패턴 수집해서 공통점 정리해줘" |
| **데이터 전처리** | 사용자 인터뷰 텍스트 분류, 피드백 태깅, 설문 결과 요약 | "인터뷰 녹취록에서 pain point만 추출해서 카테고리별로 묶어줘" |
| **베리에이션 생성** | 카피 변형, 레이아웃 옵션, 컬러 조합 생성 | "이 버튼 카피 톤별로 5가지 버전 만들어줘" |
| **스펙 문서화** | 컴포넌트 명세 작성, 레드라인 설명, 변경 이력 정리 | "이 컴포넌트 상태별 스펙을 개발자용 마크다운으로 정리해줘" |
| **QA·검수** | 디자인 가이드라인 준수 여부 체크, 접근성 검토, 일관성 검사 | "이 화면들이 우리 디자인 원칙 위반하는 부분 찾아줘" |

### 하단: 자동화 루틴 설계 원칙
- 에이전트에게 줄 수 있는 명확한 기준이 있는 일을 먼저 자동화
- 결과물은 항상 디자이너가 검토·판단
- 반복할수록 기준이 정교해지고 에이전트 퀄리티도 올라감

## 4.6 코딩 에이전트와 퀵 프로토타이핑

**도입 카피:** "코드를 몰라도 작동하는 것을 만들 수 있게 됐다"

**페이지 구성:** 상단 Before/After 2블록 + 하단 레벨별 워크플로우 차트

### 상단: Before / After 2블록 대비

**Before**
프로토타이핑의 한계가 Figma였음. 실제로 작동하는 걸 만들려면 개발자에게 요청해야 했고, 그 사이에 아이디어가 죽거나 왜곡됨. 코드는 내 영역이 아니었음.

**After**
코딩 에이전트에게 원하는 것을 말로 설명하면 작동하는 코드가 나옴. 디자이너가 직접 클릭되고 데이터가 흐르고 로직이 작동하는 프로토타입을 당일에 만들 수 있음.

### 하단: 퀵 프로토타이핑 레벨별 워크플로우 차트

4개 레벨 컬럼 가로 균등 분할. 각 컬럼은 위에서 아래로:
1. 목적 (라벨 + 설명)
2. 결과물 (라벨 + 설명)
3. 레벨 헤더 (예: L1. 인터랙티브 목업)
4. 흐름도 (도구 박스 위→아래 세로 흐름, 메인 박스는 ↓ 화살표 연결, 대안 박스는 ⋮ 점선 연결)
5. 프롬프트 예시 (보라톤 좌측 마커)

**L1 인터랙티브 목업**
- 목적: 플로우·네비게이션 검증, 이해관계자 초기 피드백
- 결과물: 클릭 가능한 하이파이 프로토타입, 공유 링크
- 흐름도: Figma ↓ Figma Make ⋮ Google Stitch (대안)
- 프롬프트 예시: "Figma로 [화면 설명]을 디자인했어. 클릭하면 다음 화면으로 넘어가는 인터랙티브 프로토타입으로 만들고 싶은데 Figma Make로 어떻게 연결하면 돼?"

**L2 작동하는 UI**
- 목적: 실제 인터랙션·로직 검증, 빠른 컨셉 확인
- 결과물: 브라우저에서 돌아가는 HTML/React, URL 공유 가능
- 흐름도: Claude 채팅 ↓ Claude Design / Artifacts ⋮ v0 / Bolt (대안)
- 프롬프트 예시: "[만들고 싶은 기능 설명]. 실제로 클릭하고 입력이 되는 HTML 화면으로 만들어줘. 디자인보다 기능 동작 확인이 목적이야."

**L3 데이터 연결**
- 목적: 엣지 케이스 확인, 실제 데이터 흐름 검증
- 결과물: API 붙인 작동 화면, 실데이터 기반 검증본
- 흐름도: Claude Code / Cursor ↓ Google AI Studio ↓ Postman / 실데이터 검증 ↓ Claude Design
- 프롬프트 예시: "[만들고 싶은 기능 설명]. 실제 API 데이터를 받아서 보여주는 프로토타입이 필요해. Claude Code로 구현해줄 수 있어? 어떻게 시작하면 돼?"

**L4 배포 가능한 PoC**
- 목적: 이해관계자 데모, 실현 가능성 검증
- 결과물: 실제 URL로 접근 가능한 작동하는 데모
- 흐름도: Claude Code ↓ Claude Design ↓ Vercel 배포
- 프롬프트 예시: "[만들고 싶은 기능 설명]. 실제로 URL로 접근 가능한 데모를 만들고 싶어. Claude Code로 구현하고 Vercel로 배포하는 전 과정을 같이 진행해줄 수 있어?"
- 주석 (L4 흐름도 마지막 박스 아래): "* 간단한 데모가 급할 땐 Lovable로 대체 가능 (코드 제어권 제한)"
