# Part 2. 에이전트 아키텍처: 사고하고 행동하는 지능의 구조

---

## 2.1 에이전트 존재의 설계도

**페이지 구성:** 봇 일러스트 중앙 배치 + 각 부위에서 라인 연결로 설명 텍스트 연결

**도입 카피:** 일러스트 하단 또는 옆에 작게 배치
"에이전트라는 봇에 스킬과 메모리를 갈아끼우면 특정 가치를 창출하는 에이전트가 된다"

### 봇 일러스트 구성
Flat한 단순 기하도형 조합 스타일 (미니멀 일러스트레이션)

### 각 부위 설명 (라인 연결)

| 부위 | 레이블 | 설명 |
|------|--------|------|
| 머리 | 두뇌 LLM | 사용자 쿼리 이해 후 "어떤 사이트에서 무엇을 클릭해야겠다"고 추론 |
| 안테나 | Perception | 가격 변동, 재고 알림, 사용자 메시지 등 외부 신호를 감지 |
| 한쪽 손 | 스킬 | "최저가를 찾을 때 배송비까지 포함해서 계산하라"는 판단 로직 부여 |
| 다른쪽 손 | 메모리 | 신용카드 정보, 배송지 주소, 선호 브랜드 등의 맥락 보유 |
| 몸 | 액추에이터 | 뇌의 명령을 받아 실제 브라우저를 열고 로그인, 클릭, 결제 수행 |

## 2.2 에이전트 액추에이터

**페이지 구성:** 상단 역할 설명 3블록 + 하단 종류 테이블

**도입 카피:** "에이전트의 추론(사고)을 물리적/디지털 행동으로 변환하는 하드웨어 및 소프트웨어적 실행 체계"

### 액추에이터의 역할 (3블록)

**1. Translation**
LLM이 내놓는 추상적 결론을 특정 통신 규격이나 구체적 디지털 신호로 번역

**2. Interface**
에이전트가 채팅창 밖 온/오프라인 세계와 접촉하는 '가상의 신체'

**3. Feedback Loop**
실행 결과를 다시 LLM에게 전달하는 신경계 역할. 에이전트가 다음 행동을 판단

### 액추에이터의 종류 (테이블, 종류/설명/예시 3컬럼)

| 종류 | 설명 | 예시 |
|------|------|------|
| GUI 액추에이터 | 사람이 보는 화면을 인식하고 마우스/키보드처럼 직접 조작 | Anthropic Computer Use, Microsoft UFO, OpenClaw* |
| 디지털 액추에이터 | API 호출·워크플로우 트리거로 디지털 서비스를 직접 제어 | Zapier, Make, n8n |
| 물리적 액추에이터 | 센서·구동 신호로 IoT·로봇 등 물리 디바이스를 제어 | IoT 제어, 로봇 제어 |

*주석: OpenClaw는 WhatsApp, Telegram 같은 메시징 플랫폼을 통해 GUI를 조작하는 로컬 오케스트레이션 레이어로, GUI 액추에이터이자 에이전트 프레임워크의 성격을 동시에 가짐

## 2.3 도구에서 스킬로

**페이지 구성:** 상단 개념 대비 + 계층 구조 시각화 / 중단 오픈 스탠다드 / 하단 생태계 비교

**도입 카피:** "단순 기능 호출에서 재사용 가능한 지식 패키지로"

### 상단: Tool vs Skill 개념 대비 (2블록 대칭, 각 블록 = 정의 + 예시)

**Tool**
에이전트가 특정 파라미터로 호출하는 단일 기능. 에이전트가 언제 어떻게 쓸지 직접 제어.
예: "환율 계산하기", "API 호출하기"

**Skill**
도메인 지식 + 판단 로직 + 워크플로우가 패키징된 상위 레벨 역량. 에이전트가 맥락을 보고 스스로 적합한 스킬을 선택해 실행.
예: "재무 분석", "최저가 쇼핑 대행"

### 중단: SKILL.md 오픈 스탠다드

Anthropic이 2025년 12월 오픈 스탠다드로 공개. Claude Code, OpenAI Codex, OpenClaw 모두 채택 — 한 번 만든 스킬을 여러 에이전트에서 사용 가능.
MCP가 에이전트의 연결 표준이라면, SKILL.md는 에이전트의 행동 표준.

### 하단: 스킬 생태계의 두 방향

| | Claude 스킬 생태계 | OpenClaw (ClawHub) |
|---|---|---|
| 성격 | 엔터프라이즈 SaaS 연동 중심 | 개인 자동화·로컬 실행 중심 |
| 주요 제공자 | Atlassian, Figma, Notion 등 공식 파트너사 | 커뮤니티 개발자 (13,700개+) |
| 적합한 용도 | 조직 단위 업무 툴 워크플로우 | 개인 디바이스·메시징 앱 기반 자동화 |
| 거버넌스 | Anthropic + 파트너사 검증 | 오픈 등록, VirusTotal 스캐닝 |

## 2.4 메모리 아키텍처: 에이전트의 기억법

**페이지 구성:** 상단 단기/장기기억 2블록 + 하단 장기기억 3유형 테이블

**도입 카피:** "에이전트는 어떻게 기억하고, 무엇을 기억하는가"

### 상단: 단기기억 vs 장기기억 (2블록)

**단기기억 — In-Context Memory**
현재 세션 안에서만 유지되는 작업 기억. 지금 처리 중인 정보, 대화 히스토리, 태스크 중간 결과. 세션이 끝나면 사라짐.

**장기기억 — External Memory**
세션이 끝나도 유지되는 기억. 외부 저장소에 저장되고 필요할 때 RAG(Retrieval-Augmented Generation) 방식으로 꺼내옴.

### 하단: 장기기억의 3가지 유형 테이블

| 유형 | 저장하는 것 | 예시 |
|------|------------|------|
| **Episodic** 경험 기억 | 과거 상호작용의 기록 | "지난 세션에서 사용자가 배송지를 강남구로 바꿨다" |
| **Semantic** 사실 기억 | 사용자 정보·선호·도메인 지식 | "이 사용자는 나이키를 선호하고 신용카드는 삼성카드다" |
| **Procedural** 절차 기억 | 반복 수행하는 워크플로우·판단 로직 | "최저가 탐색 시 배송비 포함해서 계산한다" |

*연결 주석: Procedural 메모리는 2.3에서 다룬 스킬로 구현됨 — 판단 로직을 SKILL.md로 패키징하면 재사용 가능한 절차 기억이 됨

## 2.5 OpenClaw와 Agentic AI Framework

**페이지 구성:** 상단 g2 (정의 + 공통 원칙 좌우 배치) + 하단 비교 테이블 (flex:1로 세로 공간 확보)

**도입 카피:** "에이전트를 움직이게 하는 두 가지 방식의 프레임워크"

### 상단 좌측: 에이전트 프레임워크란 무엇인가

프레임워크는 LLM의 추론을 실제 환경에서 작동시키는 실행 레이어다.

LLM 단독으로는 단일 응답을 넘어선 과업을 수행할 수 없다 — 작동 환경이 있어야 비로소 에이전트가 된다.

### 상단 우측: 두 프레임워크가 공유하는 설계 원칙

- LLM은 두뇌, 프레임워크는 신체와 환경
- 메모리·스킬·액추에이터가 조합되어야 에이전트가 완성됨
- 오케스트레이션 레이어가 추론·도구 호출·메모리 접근을 하나의 작업 흐름으로 엮는다

### 하단: OpenClaw vs Agentic AI Framework 비교 테이블

| | OpenClaw | Agentic AI Framework (AutoGen/CrewAI 계열) |
|---|---|---|
| **성격** | 상시 실행 개인 자동화 에이전트 | 멀티에이전트 오케스트레이션 프레임워크 |
| **실행 환경** | 로컬 머신, 메시징 앱 연결 | 클라우드·로컬 모두 가능, 코드 기반 배포 |
| **에이전트 구조** | 단일 에이전트 + 스킬 확장 | 역할 분담된 멀티에이전트 협업 |
| **진입장벽** | 설치 후 대화로 바로 사용 | 코드로 에이전트·태스크 설계 필요 |
| **스킬/툴** | ClawHub 커뮤니티 스킬 | 커스텀 툴·API 코드로 직접 정의 |
| **적합한 용도** | 개인 자동화, 빠른 프로토타이핑 | 복잡한 멀티스텝 업무 파이프라인 |
